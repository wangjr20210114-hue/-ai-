# 联网搜索策略与设置

## 当前执行链

1. 独立 LLM 先读取当前问题和已过滤的非敏感长期记忆，决定是否联网、合并事实查询，并判断图片是否有助于理解。规划调用默认最多占用 6 秒（可用 `CAPABILITY_PLAN_TIMEOUT_SECONDS` 在 3–12 秒内调整）；超时时不使用关键词规则，主模型继续持有完整工具集并做语义决策。
2. 一轮对话只允许执行一次 `rich_search`；同一轮参数改写会复用同一个任务。
3. 只要 LLM 语义规划确认需要联网且图片上限不为 0，就允许从同一份事实结果提取并审核候选图；回答模型仍可不采用。规划器给出的事实与视觉意图合并进一次 WSA SearchPro 请求，不再为近义图片词额外调用第二次 SearchPro。
4. 来源页媒体可并发抽取，随后最多选取 4 张候选图并发进行单图视觉审核；每张图的 Provider 降级顺序为混元 → Cloudflare Workers AI → 百炼 Qwen-VL → Gemini。广告、促销、二维码、Logo、UI、占位图和无关图不进入回答。
   WSA 新闻结果把主图放在独立图片字段或 `passage` 内嵌 `<img>` 时都会被识别；HTTP 主图统一升级为 HTTPS 后再送视觉审核，避免 Preview 的混合内容拦截。
   回答模型会等待这组有硬时间上限的审核结束，只接收审核通过的真实图片 URL，并自行在对应段落输出标准 Markdown 图片。前端不再根据段落结构猜测图片位置；视觉 Provider 缺失、失败、超时或明确拒绝时都不发布该图片。
5. 结果按规划器产出的合并查询、日期约束和搜索设置写入 LangGraph Store TTL 缓存；同义用户问法可以复用相同语义查询。Provider 超时且存在仍在安全陈旧窗口内的缓存时返回带 `stale_cache_hit` 标记的旧结果。
6. 回答不再追加独立“来源”目录。模型把 Markdown 链接放在对应事实段落内，前端显示成带标题的可点击来源；审核后的图片由同一个模型直接使用 `![说明](URL)` 编排，并随最终回答流式渲染。

## 用户可调参数

“实时搜索”由 Skills 广场统一启用或停用，取代输入框曾经存在但后端不读取的单轮“联网搜索”勾选框。关闭 Skill 后，当前模型不再获得搜索工具；重新开启即可恢复。搜索参数由 Makers Workspace 持久化，并在普通“设置 → 搜索体验”中提供。只有 Skills 广场开启“实时搜索”时才展示这一分区；关闭 Skill 不删除已保存偏好，重新开启后继续使用。

| 设置 | 可选值 | 默认 | 说明 |
| --- | --- | --- | --- |
| 网页结果上限 | 4 / 8 / 12 / 18 | 8 | 这是送入答案模型的去重结果上限，不是 Provider 计费配额 |
| 候选图片上限 | 0 / 1 / 2 / 4 | 2 | 0 表示关闭图片搜索和视觉审核 |
| 并行查找图片 | 开/关 | 开 | 开启时图片与事实检索并发，并并行抓取来源页；SearchPro 无论开关都只调用一次 |

默认 8/2/并行是当前推荐的平衡档：8 个来源足以支撑普通新闻综合；2 张图片能保持理解价值，同时把网页抓取页数和视觉审核候选控制在较小范围。深度档仍可提升到 18/4，但来源页串行提取可能触发 5 秒媒体预算，所以关闭并行只用于故障比较，不是推荐配置。

## 调用次数与性能证据

- 所有问题：每轮最多一次 `rich_search`，内部一次 WSA SearchPro；视觉词并入同一个 Query。
- 每次工具返回在 `search_config` 写入 `turn_tool_invocations` 和 `turn_provider_calls`，并输出 `rich_search turn_audit` 日志。前者用于发现模型是否重复发起工具，后者才是实际外部检索数；同轮重复调用复用同一任务，缓存命中时 Provider 数为 0。
- 视觉审核：最多 4 个单图请求并发执行；混元视觉单次请求只接收一张图，但不同图片不需要串行。每张图片的 Provider 降级链共享同一个时间预算。
- 默认硬预算：能力规划 6 秒；主模型 Makers 网关首响应 12 秒，超时后 DeepSeek 降级同样 12 秒；SearchPro 10 秒、来源页媒体 5 秒、视觉审核 7 秒。因此未命中缓存的 `rich_search` 工具目标最坏约 22–24 秒，低于 30 秒。5–10 秒目标依赖 SearchPro 在预算内较快返回、结果自带主图或缓存命中；完整最终回答还包含主模型答案生成，不能把该时间全部算作 `rich_search`。
- Preview `dp2ak2xcj97a` 的真实取证：冷查询“最近AI有什么新进展”约 14.4 秒进入搜索、23.6 秒完成工具并开始组织，完整长回答约 48.3 秒；同一问题新会话复问时，持久缓存把外部检索压到约 1 秒，完整回答约 33 秒。两次均只执行一次 SearchPro，且新闻回答保留 1 张文章主图。
- 因此已经达到“`rich_search` 工具本身最慢低于 30 秒”的目标；5–10 秒可在缓存命中，或搜索、页面提图和视觉审核都快速返回时达到，但不能承诺所有冷查询在 5–10 秒内生成完整长答案。2026-07-21 的地点线上回归发现规划器和主模型都可能在原 300 秒网络超时前长期占住请求，现已把独立规划限制为 6 秒，并把主模型主/备 Provider 的单次无响应预算分别限制为 12 秒；超时仍由模型降级或主模型继续语义路由，不取消 LLM 判断，也不增加搜索调用。
- 多地点地图推荐中，模型仍先按语义选择 2–12 个具体地点，再由位置服务逐项验证；各地点核实并行执行，每个地点默认 5 秒总预算。腾讯地点候选按查询名称覆盖度排序，官方餐馆/景点实体优先于“三里屯”等泛化区域点；同时支持“官方短名称包含在模型描述性名称中”的安全匹配，减少命中真实结果后仍等待公共降级源的情况。未匹配的腾讯候选不会在公共降级为空时重新混回结果，因此缺失餐馆不会被商圈冒充。成功核实的地点会立即写入并推送地图，未核实项返回给模型自然说明；只有全部地点均未核实时才拒绝生成地图，且绝不把未核实文本伪装成真实点位。

每次结果附带 `search_config`（含本轮工具/Provider 计数）、`cache_hit`/`stale_cache_hit`、`cache_age_seconds`、`timings_ms` 和 `vision_diagnostics`，可以区分模型工具请求数、实际 Provider 请求数、缓存命中、搜索、网页媒体和视觉审核阶段。
