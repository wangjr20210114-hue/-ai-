# 联网搜索策略与设置

## 当前执行链

1. 独立 LLM 先读取当前问题和已过滤的非敏感长期记忆，决定是否联网、合并事实查询，并判断图片是否有助于理解。
2. 一轮对话只允许执行一次 `rich_search`；同一轮参数改写会复用同一个任务。
3. 只要 LLM 语义规划确认需要联网且图片上限不为 0，就允许从同一份事实结果提取并审核候选图；回答模型仍可不采用。需要独立图片查询时，事实查询和图片查询默认并发请求 WSA SearchPro，而不是先后串行；图片布尔值偶发漏判时不再让整个图片管线失效，也不增加第二次 SearchPro 请求。
4. 网页媒体并发抽取后，最多一次批量 HY-Vision 调用审核真实像素；广告、二维码、Logo、UI、占位图和无关图不进入回答。
   WSA 新闻结果把主图放在独立图片字段或 `passage` 内嵌 `<img>` 时都会被识别；HTTP 主图统一升级为 HTTPS 后再送视觉审核，避免 Preview 的混合内容拦截。
5. 结果按问题、记忆、日期约束和搜索设置写入 LangGraph Store TTL 缓存。

## 用户可调参数

右侧“主动策略与预算 → 联网搜索设置”持久化以下参数：

| 设置 | 可选值 | 默认 | 说明 |
| --- | --- | --- | --- |
| 网页结果上限 | 4 / 8 / 12 / 18 | 8 | 这是送入答案模型的去重结果上限，不是 Provider 计费配额 |
| 回答图片上限 | 0 / 2 / 4 | 2 | 0 表示关闭图片搜索和视觉审核 |
| 事实与图片查询并行 | 开/关 | 开 | 开启时独立的两次 SearchPro 请求同时开始，响应时间取较慢者而不是两者相加 |

默认 8/2/并行是当前推荐的平衡档：8 个来源足以支撑普通新闻综合；2 张图片能保持理解价值，同时把网页抓取页数和视觉审核候选控制在较小范围。深度档仍可提升到 18/4。

## 调用次数与性能证据

- 不需要图片或图片查询与事实查询相同时：一次 WSA Provider 请求。
- 需要独立图片查询时：一次 `rich_search` 工具调用内并发两次 WSA Provider 请求；不是模型重复调用工具。
- 视觉审核：最多一次批量模型请求。
- 2026-07-20 原 Preview 的一次未缓存完整新闻流请求耗时约 54.4 秒；相同问题命中缓存后的界面复现约 22 秒并返回 2 张图片。推荐档首次完整请求曾约 76 秒且图片规划漏判；加入来源与精炼约束后第二次约 50 秒并完整返回来源，但 LLM 同时漏判图片和富回答标记，因此最终策略改为所有语义联网回合复用事实结果审核候选图。控制台聚合日志没有暴露子阶段耗时，因此不能把总耗时全部归因于搜索。
- 最终 Preview `dpxl03mjkz33` 已能解析 WSA `passage` 内嵌的腾讯新闻主图，图片 HTTPS 由浏览器直接验证可达且内容相关；完整回答约 67 秒并带来源，但仍未输出图片。项目只有 `HUNYUAN_IMAGE_API_KEY`，没有独立 `HUNYUAN_VISION_API_KEY`，因此当前结论是视觉审核授权/调用未通过，不能归因于 SearchPro 或 LLM 路由，也不能把配图用例标记为通过。

每次结果现在附带 `search_config`、`cache_hit`、`timings_ms` 和 `vision_diagnostics`，可以区分 Provider 请求数、缓存命中、搜索、网页媒体和视觉审核阶段。
